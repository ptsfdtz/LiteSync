# LiteSync FAQ（规划中）

## 1. 为什么某些文件没有及时同步？

可能原因：

- 文件系统事件存在延迟、合并或丢失。
- 文件正在被写入，触发了重试而非立即同步。
- 文件命中排除规则（`exclude`）。
- 调度器防抖窗口内尚未到执行时间。

排查建议：

1. 查看日志是否有对应 `job_id` 与 `run_id` 的记录。
2. 确认文件路径未被排除规则匹配。
3. 确认任务状态是否为 `enabled=true`。
4. 手动触发一次同步（`TriggerNow`），验证是否可补偿成功。
5. 若持续出现，检查周期校验（`periodic_reconcile`）是否启用。

## 2. 如何处理被占用文件或权限不足？

被占用文件：

- 系统会按“可重试错误”处理并进行退避重试（规划中）。
- 若多次失败，将记录为错误项，不阻塞其他文件同步。

权限不足：

- 检查源目录读权限、目标目录写权限。
- Windows 需关注 UAC 与受保护目录。
- macOS 需确认应用对相关目录的授权。
- Linux 需检查目录属主、权限位和挂载选项。

建议：

1. 避免将目标目录设为系统保护路径。
2. 先用手动复制验证当前用户是否具备读写权限。

## 3. 自启动失败如何排查？

常见原因：

- 权限不足，无法写入自启动配置。
- 目标平台机制差异导致路径或格式错误。
- 可执行文件路径变更后未更新自启动项。

排查步骤：

1. 在应用内检查 `StartupService.Status()`（规划中）。
2. 查看日志中的 `startup` 相关错误分类。
3. 按平台检查：
   - Windows：注册表 `Run` 或计划任务记录
   - macOS：`LaunchAgents` 配置是否存在且有效
   - Linux：`~/.config/autostart` 文件是否正确
4. 重新执行启用自启动并再次核对状态。

## 4. 如何安全退出并确保不丢任务？

建议流程：

1. 通过托盘或主界面的“退出”动作退出，不要强制结束进程。
2. 程序收到退出信号后：
   - 停止接收新事件
   - 尝试完成在途任务或安全中断
   - 写入最终状态与错误摘要（规划中）
3. 退出后可在日志中确认最后一次 `run_id` 结束状态。

注意：

- 强制结束进程可能导致状态未落盘，下次启动将依赖周期校验补偿。

## 5. 日志文件在哪里看？

默认路径（规划中）：

- Windows: `%APPDATA%\\LiteSync\\logs`
- macOS: `~/Library/Application Support/LiteSync/logs`
- Linux: `~/.config/litesync/logs`

如果配置了 `app.log_dir`，则优先使用自定义路径。

建议至少关注以下字段：

- `job_id`：任务标识
- `run_id`：单次同步标识
- `reason`：触发原因（startup/file_event/manual/reconcile）
- `error_code`：错误分类

## 6. 为什么目标目录里出现“冲突备份文件”？

说明：

- 当冲突策略为 `backup_then_overwrite` 时，LiteSync 会先备份目标文件，再写入源文件版本。
- 这是为了在保证一致性的同时保留可恢复性。

处理建议：

1. 对照日志确认冲突触发原因。
2. 评估是否需改用 `overwrite` 或 `skip`。
3. 定期清理冲突备份目录（规划中提供清理策略）。

## 7. 改了配置为什么没有立刻生效？

原因：

- 部分字段仅支持下次启动生效（如 `run_mode`、目录路径）。
- 热更新能力为规划中，当前以安全优先，不保证所有字段即时应用。

建议：

1. 参照 `docs/CONFIG.md` 中“配置热更新策略”表确认字段行为。
2. 对关键字段变更后，建议重启任务或重启应用。
